Где уместно, термин снабжён ссылкой на соответствующий раздел, где он разбирается подробно. В конце — список авторитетных первоисточников: спецификации, документация ядра, статьи и сайты проектов, на которые стоит опираться вместо пересказов из третьих рук.

## Базовые понятия

- **Контейнер** — изолированный процесс (или группа процессов), запущенный на общем с хостом ядре Linux, но видящий собственное пространство имён, ограниченный по ресурсам и помещённый в отдельную корневую файловую систему. Это не виртуальная машина: гипервизора и гостевого ядра нет (см. [/virtualization/containers-vs-vm/](/virtualization/containers-vs-vm/)).
- **Образ (image)** — неизменяемый артефакт, содержащий файловую систему приложения и метаданные (команда запуска, переменные окружения, точки входа), из которого создаются контейнеры. Подробно — в разделе [/containerization/images/](/containerization/images/).
- **Слой (layer)** — отдельный набор изменений файловой системы (добавленные, изменённые или удалённые файлы), идентифицируемый по содержимому. Образ — это упорядоченный стек слоёв, поверх которого при запуске контейнера добавляется тонкий записываемый слой.
- **OverlayFS** — объединяющая (union) файловая система ядра Linux, представляющая несколько каталогов (нижние слои `lowerdir` только для чтения и верхний `upperdir` для записи) как единое дерево. Штатный storage-драйвер современного Docker и containerd (`overlay2`).
- **Copy-on-write (CoW)** — стратегия, при которой данные физически копируются только в момент первой записи: пока контейнер лишь читает файл из нижнего слоя, копии не создаётся. Экономит место и ускоряет запуск контейнеров из общего образа.
- **Whiteout** — специальная запись в верхнем слое OverlayFS (символьный файл устройства с номерами 0/0), помечающая файл из нижнего слоя как удалённый, хотя физически в нижнем слое он остаётся.

## Изоляция: namespaces

- **Namespace (пространство имён)** — механизм ядра Linux, который ограничивает и изолирует видимость глобальных ресурсов для группы процессов, заставляя их «думать», что они владеют ресурсом в одиночку. Основа изоляции контейнеров — см. [/containerization/namespaces/](/containerization/namespaces/).
- **Типы namespaces** — `mnt` (точки монтирования), `pid` (идентификаторы процессов), `net` (сетевой стек: интерфейсы, таблицы маршрутизации, порты), `ipc` (System V IPC и очереди POSIX), `uts` (hostname и domainname), `user` (отображение UID/GID), `cgroup` (видимость корня иерархии cgroup), `time` (смещение системных часов).
- **User namespace** — namespace, отображающий идентификаторы пользователей и групп: процесс может быть root (UID 0) внутри namespace, оставаясь непривилегированным пользователем на хосте. Ключевой механизм для rootless-контейнеров.

## Лимиты ресурсов: cgroups

- **cgroups (control groups)** — подсистема ядра Linux для учёта, ограничения и приоритизации потребления ресурсов (CPU, память, I/O, PID) группами процессов. Разбор — в разделе [/containerization/cgroups/](/containerization/cgroups/).
- **cgroups v1** — первоначальная реализация с отдельной иерархией для каждого контроллера; гибкая, но сложная и местами несогласованная (например, разрозненный учёт памяти и swap).
- **cgroups v2** — переработанная унифицированная иерархия с единым деревом для всех контроллеров и согласованной моделью (`cgroup.controllers`, `cgroup.subtree_control`). Используется по умолчанию в современных дистрибутивах.
- **Контроллер cgroup (controller / subsystem)** — компонент, управляющий конкретным видом ресурса: `cpu`, `cpuset`, `memory`, `io`, `pids`, `hugetlb` и другие. Контроллеры включаются в узлах дерева и применяют свои лимиты к процессам группы.
- **OOM killer** — механизм ядра, принудительно завершающий процессы при исчерпании памяти. Для cgroup с лимитом памяти срабатывает локально внутри группы: убивается процесс контейнера, а не произвольный процесс хоста.

## Безопасность

- **Linux capability** — единица дробления привилегий root: вместо «всё или ничего» право выполнять привилегированную операцию (например, `CAP_NET_ADMIN`, `CAP_SYS_ADMIN`) выдаётся точечно. Среды выполнения по умолчанию урезают набор capabilities контейнера — см. [/containerization/security/](/containerization/security/).
- **seccomp** — фильтр системных вызовов (secure computing mode), позволяющий разрешить или запретить отдельные syscalls для процесса с помощью BPF-программы. Docker и Kubernetes применяют профили seccomp, сокращающие доступную контейнеру поверхность ядра.
- **AppArmor** — система мандатного контроля доступа (MAC) в Linux, ограничивающая возможности программ по путям файлов и набору операций через профили. Применяется к контейнерам по имени профиля.
- **SELinux** — система мандатного контроля доступа на основе меток (labels) и политик, разделяющая процессы и файлы по контекстам безопасности. Распространена в дистрибутивах семейства RHEL для изоляции контейнеров.

## Стандарты и среды выполнения (OCI)

- **OCI (Open Container Initiative)** — организация под эгидой Linux Foundation, разрабатывающая открытые стандарты для контейнеров. Подробно — в разделе [/containerization/runtimes/](/containerization/runtimes/).
- **OCI image-spec** — спецификация формата образа: структура манифеста, конфигурации и слоёв, правила адресации по digest.
- **OCI runtime-spec** — спецификация среды выполнения: формат OCI bundle и команды жизненного цикла контейнера (create/start/kill/delete).
- **OCI distribution-spec** — спецификация HTTP API реестра для загрузки и выгрузки образов (push/pull), стандартизирующая работу с registry.
- **OCI bundle** — каталог на диске, содержащий файл `config.json` (конфигурация по runtime-spec) и `rootfs` (корневую файловую систему). Именно его получает на вход низкоуровневая среда выполнения.
- **runc** — эталонная низкоуровневая среда выполнения OCI на Go: создаёт namespaces и cgroups, применяет настройки безопасности и запускает процесс контейнера из bundle.
- **crun** — альтернативная низкоуровневая среда выполнения на C: быстрее и легче runc, с полной поддержкой cgroups v2; часто используется в связке с CRI-O.
- **containerd** — высокоуровневая среда выполнения (демон), управляющая полным жизненным циклом контейнеров: образами, снапшотами, сетью и вызовом runc. Проект CNCF, ядро Docker Engine.
- **CRI-O** — легковесная высокоуровневая среда выполнения, спроектированная специально под Kubernetes и реализующая интерфейс CRI без лишней функциональности.
- **shim** — процесс-посредник между средой выполнения (containerd/CRI-O) и процессом контейнера. Он позволяет родительскому демону перезапускаться и обновляться без остановки уже работающих контейнеров и удерживает их stdio и код возврата.

## Интерфейсы Kubernetes

- **CRI (Container Runtime Interface)** — gRPC-интерфейс между kubelet и средой выполнения контейнеров, благодаря которому Kubernetes не привязан к конкретному runtime (containerd, CRI-O).
- **CNI (Container Network Interface)** — стандарт и набор плагинов для настройки сети контейнеров: выделение интерфейсов, IP-адресов и маршрутов при создании Pod. См. [/containerization/networking/](/containerization/networking/).
- **CSI (Container Storage Interface)** — стандартный интерфейс для подключения систем хранения к оркестраторам, позволяющий поставщикам storage писать драйверы независимо от ядра Kubernetes. См. [/containerization/storage/](/containerization/storage/).

## Реестры и сборка образов

- **Реестр (registry)** — сервис хранения и распространения образов (Docker Hub, GitHub Container Registry, Harbor), работающий по OCI distribution-spec.
- **Digest** — криптографический хеш (обычно SHA-256) содержимого манифеста или слоя, дающий неизменяемый и проверяемый идентификатор образа (`sha256:...`), в отличие от подвижного тега.
- **Манифест (manifest)** — JSON-документ, описывающий образ: ссылки на конфигурацию и слои по digest, их размеры и типы. Может быть мультиплатформенным (manifest list / image index).
- **Dockerfile** — текстовый файл с инструкциями (`FROM`, `RUN`, `COPY`, `CMD` и др.), по которому собирается образ. Практика — в разделе [/containerization/docker/](/containerization/docker/).
- **Кэш слоёв (layer cache)** — повторное использование результатов ранее выполненных инструкций Dockerfile: если инструкция и её контекст не изменились, слой берётся из кэша вместо пересборки.
- **Multi-stage build** — сборка в несколько стадий `FROM`, где артефакты из стадии-сборщика копируются в финальный минимальный образ, а инструменты сборки в результат не попадают.
- **Distroless** — минималистичные образы без пакетного менеджера и оболочки, содержащие только приложение и его runtime-зависимости; уменьшают размер и поверхность атаки.

## Хранение и сеть

- **Volume** — управляемый средой выполнения механизм постоянного хранения данных контейнера вне его записываемого слоя; переживает удаление контейнера. См. [/containerization/storage/](/containerization/storage/).
- **Bind mount** — прямое монтирование произвольного каталога или файла хоста внутрь контейнера; данные и права полностью контролируются хостовой ФС.
- **tmpfs** — монтирование в оперативную память: данные существуют только во время работы контейнера и не попадают на диск, что удобно для секретов и временных файлов.
- **Storage-драйвер** — компонент среды выполнения, отвечающий за то, как слои образа и записываемый слой объединяются на диске (`overlay2`, `btrfs`, `zfs`, `devicemapper`).
- **veth (virtual ethernet)** — пара виртуальных сетевых интерфейсов, работающая как «кабель»: один конец в сетевом namespace контейнера, другой — на хосте. Базовый кирпич сети контейнеров — см. [/containerization/networking/](/containerization/networking/).
- **bridge** — программный коммутатор L2 на хосте (например, `docker0`), к которому подключаются хостовые концы veth, обеспечивая связь контейнеров между собой; выход во внешнюю сеть идёт через NAT хоста (правила iptables).
- **Overlay-сеть** — сеть, объединяющая контейнеры на разных хостах через инкапсуляцию пакетов (обычно VXLAN), создавая иллюзию единого L2-сегмента поверх физической сети.
- **Rootless** — режим запуска контейнеров без привилегий root на хосте, опирающийся на user namespaces; снижает риски при компрометации среды выполнения.

## Оркестрация и Kubernetes

- **Pod** — наименьшая разворачиваемая единица в Kubernetes: один или несколько контейнеров с общими namespaces (сеть, IPC) и томами. Подробно — в разделе [/containerization/orchestration/](/containerization/orchestration/).
- **ReplicaSet** — контроллер, поддерживающий заданное число одинаковых реплик Pod; при падении Pod создаёт новый.
- **Deployment** — контроллер поверх ReplicaSet, управляющий декларативными обновлениями приложения: rolling update, откаты и история ревизий.
- **Service** — стабильная абстракция доступа к набору Pod по фиксированному имени и виртуальному IP с балансировкой, скрывающая эфемерность отдельных Pod.
- **ConfigMap** — объект для хранения нечувствительной конфигурации (пары ключ-значение, файлы), монтируемой в Pod как переменные окружения или файлы.
- **Secret** — объект для конфиденциальных данных (пароли, токены, ключи); по содержимому похож на ConfigMap, но обрабатывается с дополнительными мерами (кодирование base64, ограничения доступа, опциональное шифрование в etcd).
- **kubelet** — агент на каждом узле кластера, запускающий и контролирующий Pod через CRI и докладывающий их состояние в control plane.
- **kube-apiserver** — центральный компонент control plane: единая точка входа REST API, через которую проходят все операции и обращения к хранилищу состояния.
- **etcd** — распределённое key-value хранилище на базе Raft, где Kubernetes держит всё состояние кластера; единственный источник истины.
- **Scheduler (kube-scheduler)** — компонент control plane, назначающий новым Pod подходящие узлы с учётом ресурсов, ограничений и политик размещения.
- **Control plane** — управляющий слой кластера (apiserver, scheduler, controller-manager, etcd), принимающий решения о желаемом состоянии; рабочие узлы лишь исполняют его указания.
- **Reconciliation loop (цикл согласования)** — основной принцип работы контроллеров Kubernetes: непрерывное сравнение желаемого состояния (spec) с фактическим (status) и выполнение действий для их сближения.

### Ядро Linux: namespaces и cgroups

- man-страница [namespaces(7)](https://man7.org/linux/man-pages/man7/namespaces.7.html) — обзор всех типов пространств имён.
- man-страница [cgroups(7)](https://man7.org/linux/man-pages/man7/cgroups.7.html) — модель cgroups v1 и v2, контроллеры.
- man-страница [user_namespaces(7)](https://man7.org/linux/man-pages/man7/user_namespaces.7.html) — отображение UID/GID и привилегии.
- man-страница [unshare(1)](https://man7.org/linux/man-pages/man1/unshare.1.html) — запуск процессов в новых namespaces из командной строки.
- [Control Group v2 — документация ядра Linux](https://docs.kernel.org/admin-guide/cgroup-v2.html) — авторитетное описание унифицированной иерархии и интерфейсов контроллеров.
- [Namespaces in operation — серия статей Майкла Керриска на LWN.net](https://lwn.net/Articles/531114/) — подробный разбор каждого типа namespace с примерами кода.
